# アプリ自動開発パイプライン

ユーザーから受け取ったアイデア（一言〜数行）をもとに、
設計から実装・テストまでを一気通貫で自動実行する。

各フェーズ完了時にユーザーのレビューを受け、承認後に次フェーズへ進む。

---

## Phase 0: 環境セットアップ（自動・無言で実行）

以下を順番に確認・実行する。ユーザーへの報告は不要。

1. git リポジトリが初期化されているか確認する：
   ```bash
   git status
   ```
2. 初期化されていない場合（`not a git repository` エラー）、自動で初期化する：
   ```bash
   git init
   git add .
   git commit -m "initial: project template"
   ```
3. そのまま Phase 1 へ進む。

---

## Phase 1: ヒアリング（最小限）

ユーザーのアイデアが曖昧な場合のみ、**最大3問**で補足質問する。
以下が分かっていれば質問不要：
- 何をするアプリか
- 誰が使うか
- プラットフォーム（Web / Mobile / Desktop）

分かっていれば質問せずに Phase 2 へ進む。

---

## Phase 2: コンセプト生成

ユーザーのアイデアから `docs/concept/CONCEPT.md` を自動生成する。

```
# App Concept
## Name: [アプリ名を提案]
## Concept: [一言コンセプト]
## Description: [2〜3文の説明]
## Target Users: [ターゲット]
## Use Cases: [主な利用シーン]
## Design Philosophy: [設計思想]
## Constraints: [MVPでやらないこと]
```

**→ ユーザーに提示して確認を得る**
「このコンセプトで進めてよいですか？修正点があればお伝えください。」

---

## Phase 3: 仕様生成

承認されたコンセプトをもとに `docs/spec/SPEC.md` を自動生成する。

以下を含めること：
- MVP機能一覧（最小限に絞る）
- データモデル
- 画面一覧と基本レイアウト
- 技術スタック（ユーザーの指定があればそれに従う）
- レイヤー構造（UI / Logic / Data）
- 制約事項

**→ ユーザーに提示して確認を得る**
「この仕様で進めてよいですか？追加・削除したい機能があればお伝えください。」

---

## Phase 4: 設計 & タスク生成

承認された仕様をもとに以下を自動生成する：

1. **ARCHITECTURE.md** — ディレクトリ構造、レイヤー設計、インターフェース定義
2. **TASKS.md** — フェーズ別タスク一覧（受け入れ基準付き）
3. **PROJECT_STATE.md** — プロジェクト状態

**→ ユーザーに提示して確認を得る**
「この設計とタスクで実装に進んでよいですか？」

---

## Phase 5: 実装

承認されたタスクを順番に実行する。

### 実装ルール
- TASKS.md の順番に従う
- 各タスク完了後に TASKS.md のステータスを更新する
- `codebase-awareness` スキルに従い、既存コードの存在を確認してから書く
- `ui-design` スキルに従い、モダンで時代に左右されないUIを実装する
- `.agent/rules/` のすべてのルールを遵守する

### 進捗報告
- **5タスクごと**、または**フェーズ切り替え時**に進捗を報告する
- 報告内容：完了タスク / 進行中タスク / ブロッカー（あれば）

### ブロッカー発生時
- 実装を停止し、ユーザーに報告する
- 解決策を提案した上で判断を待つ

---

## Phase 6: テスト & 検証

全機能の実装完了後：

1. ユニットテストを作成・実行する
2. ブラウザ統合でUIの動作確認を行う（Web / Electronの場合）
3. 全テスト結果を報告する

```
## テスト結果
- ユニットテスト: ✅ PASS (N/N)
- UI動作確認: ✅ PASS / ❌ FAIL（詳細）
- 未テスト項目: [一覧]
```

**→ ユーザーに最終報告する**
「MVPの実装とテストが完了しました。確認をお願いします。」

---

## Phase 7: 完了

1. PROJECT_STATE.md を最終状態に更新する
2. TASKS.md の全タスクが ✅ であることを確認する
3. 次のステップ（機能追加、デプロイ準備等）を提案する

---

## 重要事項

### フェーズゲート
- **Phase 2, 3, 4 は必ずユーザーの承認を待つ**
- 承認なしに次フェーズへ進んではならない
- ユーザーから修正指示があれば、反映してから再度確認する

### スコープ管理
- MVP機能以外は実装しない
- 「あったら便利」な機能はConstraintsに明記して除外する
- ユーザーが追加を希望した場合のみ対応する
